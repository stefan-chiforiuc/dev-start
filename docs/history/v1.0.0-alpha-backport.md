# Backport of v1.1–v1.4 into 1.0.0-alpha (historical execution plan)

> **Historical document.** This is the execution plan that was used to fold
> the originally-planned v1.1, v1.2, v1.3, and v1.4 work into a single
> first-published release. The plan still reads as if v1.0 had already
> shipped — at the time it was written, that was the assumption. In reality
> v1.0 never reached NuGet (see `RELEASING.md` § "ghost 1.0.0"), and the
> consolidation became `1.0.0-alpha` rather than `1.1.0`. For the *decision*
> and rationale, see [ADR
> 0009](../adr/0009-collapse-v1.1-v1.4-into-v1.0.0-alpha.md). This file is
> retained for traceability of what was actually built and in what order;
> some references below (`1.1.0`, "v1.0 shipped", `version.txt`) are stale
> by design and should be read as historical context, not current state.

## Context

DevStart v1.0 shipped a .NET-only scaffolder: embedded-resource capabilities
discovered by `Capability.AvailableNames()`, a `Planner` that composes them
with three platform bundles (`platform/compose`, `platform/claude`,
`platform/devcontainer`), and a `Baselines`-driven 3-way upgrade. The
`ROADMAP.md` carves v1.1 (k8s + `promote`), v1.2 (TypeScript/Fastify stack),
v1.3 (React + TanStack frontend), and v1.4 (org-level `policy` bundles) as
separate milestones.

The user wants everything from v1.1 through v1.4 folded into a single v1
delivery — full depth, full TS capability parity, in-repo policy bundles.
Delivered as release `1.1.0` (minor bump, no breaking CLI surface).
Existing v1.0 projects migrate `.devstart.json` from schemaVersion 1 → 2 on
first upgrade.

The risk surface concentrates in two places:
1. Making `Manifest.Stack` (today hardcoded `"dotnet-api"`) actually branch
   the Planner, NewCommand defaults, and Claude briefing.
2. Refactoring the hardcoded `Planner.WriteMcpConfig` if-blocks into a
   declarative `mcp` section in `capability.json` so new capabilities add MCP
   servers by data, not code.

## Key decisions (resolved)

1. **TS capabilities use `ts-` prefix** (not a branch inside shared capability folders). One folder = one slice. `AddCommand` validates prefix vs `manifest.Stack`. Document as ADR 0008.
2. **`frontend` is a single capability** with `dependsOnByStack`, not two (`ts-frontend` + `dotnet-frontend`). The React app doesn't care which API it hits.
3. **JSON fragments use a new `json-merge` injector mode** backed by `JsonMerger.cs` — necessary because `package.json` and `tsconfig.json` don't allow comment markers. Text-mode injectors remain the default for source files.
4. **`promote` writes `values.generated.yaml` by default**; `--render` shells out to `helm template` for opt-in full rendering. Keeps CI green without helm installed.
5. **`policy remove` is manifest-only** — cannot safely reverse injected fragments; prints affected files for manual cleanup. Documented limitation.
6. **MCP declarative refactor is behaviour-preserving** — verified by byte-identity snapshot test of `.mcp.json` against a v1.0 golden file.
7. **Skills folder split is non-destructive for existing projects** — `Upgrader` classifies removed-from-template paths as preserve-user-copy (Upgrader.cs:82). Called out in CHANGELOG.

## Verification

**Unit + shape tests**
```
cd /home/user/dev-start
dotnet test
```
Expect roughly double the test count; `GeneratedSourceShapeTests` green on .NET variations plus k8s + frontend; `TsGeneratedSourceShapeTests` green on TS variations; `McpDeclarativeTests` byte-identical to golden.

**End-to-end — .NET + k8s + policy**
```
dotnet pack src/DevStart.Cli -o /tmp/nupkg
dotnet tool install -g --add-source /tmp/nupkg DevStart --version 1.1.0
cd /tmp && dev-start new demo-dotnet --with postgres auth otel --deploy fly
cd demo-dotnet && just bootstrap && just up
dev-start add k8s && test -f k8s/helm/Chart.yaml && test -f k8s/kustomize/overlays/dev/kustomization.yaml
dev-start promote dev && grep -q "replicas: 1" k8s/overlays/dev/values.generated.yaml
dev-start policy list
dev-start policy apply default-open-source && grep -q default-open-source .devstart.json
dev-start policy validate
```

```
cd /tmp && dev-start new demo-ts --stack typescript --with postgres ts-auth ts-otel ts-sdk frontend
cd demo-ts
test -f apps/api/src/app.ts && test -f apps/web/src/main.tsx && test -f packages/sdk/package.json
```

**Pack-smoke CI job extension**
Extend the existing pack-smoke CI matrix with a `stack: [dotnet, typescript]` axis. For each stack, scaffold → build → test from cold. Then `dev-start add k8s`, `dev-start promote dev`, `dev-start policy apply default-open-source`, `dev-start policy validate` — exit 0 on all.
